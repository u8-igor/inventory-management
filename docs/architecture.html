System Architecture

Factory Inventory Management System — Full-stack demo with Vue 3 frontend and Python FastAPI backend

Tech Stack
Frontend
Client — Port 3000
  • Vue 3 + Composition API
  • Vue Router (SPA)
  • Vite (dev server + build)
  • Axios (HTTP client)
  • Custom SVG charts
  • CSS Grid layouts
  • i18n (English / Japanese)
Backend
Server — Port 8001
  • Python FastAPI
  • Pydantic v2 (validation)
  • Uvicorn (ASGI server)
  • In-memory data (JSON files)
  • CORS enabled (all origins)
  • Auto-generated /docs (Swagger)
Testing & Data
Quality & Persistence
  • pytest + FastAPI TestClient
  • JSON flat files (no DB)
  • Data loaded at server startup
  • Changes are not persisted
  • 32 inventory items
  • 100+ orders (full year 2025)
Request Data Flow
1 — UI
FilterBar.vue
User selects Time Period, Warehouse, Category, or Status. Reactive refs update in useFilters() singleton.
→
2 — Composable
useFilters.js
Shared singleton state. Watchers in each view detect changes and call loadData(). Maps period → month, location → warehouse.
→
3 — API Client
api.js (Axios)
Builds query string from filter object. Skips params with value "all". Sends GET to localhost:8001/api/...
→
4 — Backend
FastAPI main.py
Receives query params. Applies apply_filters() and filter_by_month() on in-memory lists. Returns Pydantic-validated JSON.
→
5 — Render
Vue View
Stores raw data in refs. Computed properties derive charts, KPIs, and tables. Template re-renders reactively.

Filter System
Time Period

Supports direct month or quarter (Q1-2025)

Warehouse
all San Francisco London Tokyo

Case-sensitive exact match

Category
all Circuit Boards Sensors Actuators Controllers Power Supplies

Case-insensitive comparison

Order Status
all Delivered Shipped Processing Backordered

Not applied to inventory endpoint

API Endpoints
Endpoint Filters Supported Description
GET /api/dashboard/summary warehouse category status month KPI aggregates — inventory value, low stock, pending orders, backlog count
GET /api/inventory warehouse category All inventory items (no time dimension)
GET /api/inventory/{id} — Single inventory item by ID
GET /api/orders warehouse category status month All orders with full filter support
GET /api/orders/{id} — Single order by ID with item breakdown
GET /api/demand — Demand forecasts — current vs forecasted, trend direction
GET /api/backlog — Backlog items — annotated with has_purchase_order flag
GET /api/spending/summary — Total procurement, operational, labor, overhead costs
GET /api/spending/monthly — Monthly cost breakdown (Jan–Dec)
GET /api/spending/categories — Spending by category with % change
GET /api/spending/transactions — Recent transactions (56 items, Sep 2025)
POST /api/purchase-orders — Create a purchase order for a backlog item
GET /api/reports/quarterly — Quarterly performance (Q1–Q4 2025)
GET /api/reports/monthly-trends — Month-over-month revenue and order trends

Key Design Decisions
Frontend Patterns
  • Filter state is a singleton — one source of truth across all views
  • Raw data in refs, derived data in computed properties (no duplication)
  • All API calls centralized in api.js — views never call fetch directly
  • Watchers on filter state trigger data reloads per view
  • Modals are lazily rendered (v-if, not v-show)
Backend Patterns
  • Data loaded once at startup — fast reads, no DB latency
  • Filtering never mutates original lists — always filters copies
  • All responses typed with Pydantic — guarantees contract with frontend
  • "all" filter value skips that filter (no explicit null handling needed)
  • Case-insensitive matching for category and status filters
